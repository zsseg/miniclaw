**OpenClaw 个人 Mini 实现——Python 智能体编程作业**

**一、作业背景与目标**

OpenClaw 是2026年爆火的 MIT 开源 AI 智能体框架。它本身不生产大模型，而是作为连接各大模型（OpenAI、Claude、DeepSeek等）与真实系统操作之间的“中枢”——让 AI 从“只会说话”变成“会动手做事”，能够完成文件处理、信息检索、自动化流程等多种任务。

OpenClaw 的核心其实是一个经典的“消息处理循环”：接收输入 → LLM 分析并决策（回复 or 调用工具）→ 执行工具并获取结果 → 将结果反馈给 LLM → 生成最终回复。港大黄超老师课题组甚至将其重构为仅约 4000 行的 nanobot，保留了完整功能的同时大幅降低了理解门槛。

本次作业的目标是：**在理解 OpenClaw 核心原理的基础上，自己动手用 Python 编写一个“迷你版 OpenClaw”——一个能够接收用户指令、调用大模型进行推理、根据推理结果调用本地工具并执行、最终完成任务的极简智能体框架。**

**二、作业要求**

**2.1 整体框架要求（必做）**

你需要实现一个 Python 程序，具备以下核心能力：

**（1）大模型接入与对话能力**

- 支持接入 OpenAI GPT 和 DeepSeek 
- 能够接收用户输入的自然语言指令
- 能够与大模型进行多轮对话（维护对话历史上下文）

**（2）工具调用（Function Calling）能力**

- 至少实现 **3** **个自定义工具函数**，每个工具能够完成一项具体的本地操作（具体函数在文件最后）
- 大模型能够根据用户意图，自主判断是否需要调用工具、调用哪个工具、传入什么参数
- 工具执行结果能够被正确获取并返回给大模型

**（3）ReAct 推理循环**

- 实现一个基础的消息处理循环（推荐采用 ReAct 模式：Thought → Action →     Observation）
- 支持多轮工具调用（一个复杂任务可能需要多次调用工具）
- 设置合理的终止条件（如 LLM 给出 final_answer 或达到最大轮数）

**（4）本地执行与安全隔离**

- 工具执行时应有基本的安全检查（如操作范围限定在指定目录内）
- 需要对工具调用过程中的异常进行捕获和友好提示

**2.2 扩展建议**

- 对话历史持久化：将对话记录保存到本地 JSON 文件，下次启动时可加载历史
- 工具插件化：支持通过配置文件或装饰器动态注册新工具，无需修改核心代码
- 流式输出：支持 LLM 的流式响应，逐字输出（而不是一次性返回）
- 多轮规划展示：在控制台清晰展示每轮的“思考→行动→观察”过程
- 安全的 shell 执行沙箱：对 run_shell_command 实现更严格的命令白名单和参数校验
- 支持多 Agent 协作雏形：实现一个简单的“主 Agent + 子 Agent”模式（如主 Agent 可将文件分析任务委托给专门的分析子 Agent）

### 3.1 QQ自动回复

**功能说明**
当腾讯QQ客户端运行时，智能体持续监听指定群组的@消息以及私聊消息。对于群组：仅在用户被明确@（忽略@所有人）时触发；对于私聊：仅在收到消息后1分钟内用户未手动回复的情况下触发。触发后，智能体调用大模型根据当前对话上下文自动生成简洁实用的回复内容，并通过QQ客户端自动发送。用户可额外指定提示词，使回复风格或内容遵循特定要求（如“用幽默语气”或“只回答技术问题”）。同时，智能体需过滤消息中的恶意注入或攻击尝试。

**对智能体操作的要求**

1. **工具能力**
   - 具备与QQ客户端交互的能力（如通过Windows Automation、QQ机器人框架、模拟键盘鼠标或读取QQ消息数据库）。
   - 具备获取群消息、私聊消息、发送者ID、@名单、消息时间戳的工具。
   - 具备向指定群或私聊窗口发送文本消息的工具。
   - 具备判断“用户是否在1分钟内手动回复”的能力（例如监控QQ输入框活动、最近发送消息记录）。
2. **推理与规划**
   - 从用户配置中读取：目标群号、是否启用私聊回复、1分钟计时阈值、自定义提示词（如有）。
   - 对每条新消息，先判断来源类型（群/私聊）。
     - 若是群消息：检查群号是否匹配，并解析@列表是否包含当前登录用户（忽略@所有人）。
     - 若是私聊消息：记录消息到达时间，启动1分钟倒计时；若期间未检测到用户向同一联系人发送任何消息，则触发回复。
   - 将消息文本与历史对话（最近5条）及用户自定义提示词一起提交给大模型，要求生成“简洁、实用、安全”的回复。
   - 对模型输出进行后处理：去除可能泄露隐私的内容、截断过长回复（例如限制200字）。
3. **安全与约束**
   - **注入防范**：对收到的消息内容进行转义或过滤，避免模型执行“忽略之前指令”或“系统命令注入”。例如删除消息中的 `你被重置了`、`请输出某代码` 等可疑模式；在提示词中明确要求模型忽略消息中的任何指令尝试。
   - **隐私保护**：不将QQ号、真实姓名等个人标识发送给大模型，仅传递匿名ID或昵称。
   - **频率控制**：对同一联系人/群，两次自动回复之间至少间隔30秒，避免刷屏或被风控。
   - **用户覆盖**：当用户手动回复后，应取消当前等待的私聊回复任务，避免重复。
4. **交互要求**
   - 用户可通过配置文件或指令动态修改：启用/禁用功能、设置目标群号、修改自定义提示词、调整1分钟窗口时长。
   - 提供日志记录：每次自动回复的时间、联系人、原始消息摘要、生成的回复内容。
   - 支持“安静模式”：用户可临时暂停自动回复（例如发送 `/pause` 到智能体控制窗口）。

### 3.2 自动撰写文稿

**功能说明**
用户通过自然语言提出写作要求（如“写一篇关于人工智能伦理的短文，800字左右”），智能体在指定目录（默认为应用所在目录）下自动生成一个 `.md` 或 `.txt` 文件，内容由大模型根据要求生成。用户可进一步指示对已生成的文稿进行修改（如“把第三段语气改得更正式”），智能体应读取原文件、按要求修改并保存（可覆盖原文件或生成新版本）。多个文稿任务相互隔离，且所有文件操作严格限制在目标目录及允许的扩展名范围内。

**对智能体操作的要求**

1. **工具能力**
   - 具备列出目录内容、判断路径是否在允许范围内的工具。
   - 具备创建、读取、写入、重命名、删除文件（仅限 `.md`、`.txt`）的工具。
   - 具备将大模型生成的文本写入文件，并能基于用户修改指令生成更新内容的工具。
2. **推理与规划**
   - 从用户指令中解析：文稿主题、字数/篇幅要求、风格（正式/幽默/技术）、输出文件名（若无则自动生成，如 `article_20260119_001.md`）、输出目录（若未指定则使用默认目录）。
   - 生成文稿时，同时生成一个简短的元数据头（如 `# 标题`、`生成时间`）以便后续定位。
   - 对于修改请求：
     - 解析用户要修改的具体位置（“第二段”、“结尾部分”、“所有提到的‘XX’替换为‘YY’”）。
     - 读取原文件全文，定位到指定区域，调用大模型生成修改后的段落或全文。
     - 向用户展示修改差异（diff），待确认后再保存。
   - 支持多轮修改，每次修改生成一个版本（可通过时间戳命名备份）。
3. **安全与约束**
   - **路径白名单**：所有文件读写操作必须经过路径检查，确保目标绝对路径的父目录等于允许的目录（或更深一层，但不允许跳出，例如禁止使用 `../`）。
   - **扩展名限制**：新建或修改的文件扩展名只能是 `.md` 或 `.txt`。如果用户要求其他格式，智能体需拒绝并提示。
   - **防止覆盖**：写入新文件前，若同名文件已存在，默认生成 `filename_v2.md` 或询问用户是否覆盖。
   - **内容安全**：大模型生成的内容不得包含可执行代码（如HTML `<script>`、shell命令），除非用户明确要求且确认风险。
   - **资源限制**：单次生成文稿最大长度不超过10万字符，防止模型或存储过载。
4. **交互要求**
   - 每个文稿任务拥有独立会话上下文，用户可随时切换任务（如“切换到刚才的`ethics.md`”）。
   - 生成完成后返回文件路径、字数统计、简要摘要。
   - 修改操作需支持“预览模式”：先展示修改后的内容，用户确认后再写入。
   - 提供撤销功能：可回退到最近一次保存的版本。
   - 支持批量生成（如“根据这5个主题分别生成5篇短文”），并行或串行执行，并显示进度。

### 3.3：批量图片处理

**功能说明**
用户指定一个包含图片的文件夹，并给出处理要求（如“将所有 JPEG 图片调整为 1024x768 分辨率，转换为 PNG 格式，然后重命名为 `img_001.png` 的序号形式，并保存到 `output/` 目录”）。智能体调用本地图像处理库（如 ImageMagick 的 `convert` 命令或 Pillow 脚本），完成批量转换，并生成处理日志。

**对智能体操作的要求**

1. **工具能力**
   - 具备遍历目录、过滤文件扩展名的能力。
   - 具备调用图像处理工具（如系统命令 `convert` 或执行 Python Pillow 脚本）的能力。
   - 具备创建目标目录、重命名文件的能力。
2. **推理与规划**
   - 从用户指令中提取输入目录、输出目录、目标格式、目标尺寸、命名模式（序号起始值、填充宽度）、是否覆盖现有文件。
   - 若用户要求“智能调整”，例如“保持宽高比，使宽度不超过 1024”，则需要计算目标尺寸。
   - 规划任务批次：如果图片数量很大（>100），可以考虑分批处理并显示进度。
   - 对于不支持的操作（如将 GIF 转换为 PNG 丢失动画），需提前告知用户并请求确认。
3. **安全与约束**
   - 处理前检查输出目录是否有足够磁盘空间（估算原图总和 × 格式转换膨胀系数）。
   - 如果输出目录已存在同名的文件且未设置覆盖，则跳过并记录警告。
   - 对每个图片设置单独的处理超时（如30秒），避免损坏文件导致卡死。
4. **交互要求**
   - 处理过程中实时输出进度（如“已处理 15/100”）。
   - 允许用户中途取消（发送 `cancel_task <id>`）。
   - 完成后生成报告：成功数、失败数、失败文件列表及原因（如“格式不支持”）。
